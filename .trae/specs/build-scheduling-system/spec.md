# 中小学走班排课系统 Spec

## Why
中小学排课工作复杂繁琐，尤其是走班制场景下教师、教室、班级之间的冲突难以人工排查。现有排课工具要么功能单一、要么操作门槛高，缺乏一套面向中小学、支持精细化控制、零基础上手的现代化排课系统。

## What Changes
- 构建一套完整的 B/S 架构中小学走班排课系统，支持 Docker 一键部署
- 实现多租户（多学校）隔离，支持超级管理员、学校账户、教师账户三级权限体系
- 实现流程式排课指引，支持拖动排课、一键排课、固定排课，并实时检测冲突
- 实现按年级精细化控制早读/正课/晚自习的节数、时间、排课时段
- 实现学期管理、课表保存、课表导出（教师课表、班级课表，支持批量与自定义）
- 实现基础数据 Excel 批量导入导出
- 实现 SQLite（默认）/ MySQL 双数据库设计，超级管理员可切换
- 实现完善的日志系统（仅超级管理员可见）
- 实现系统备份与还原（仅超级管理员）
- 实现账户申请审核流程、学校账户数据修改审批流程

## Impact
- Affected specs: 全新系统，无已有 spec 受影响
- Affected code: 全新项目，技术栈为 React + TypeScript（前端）、Python FastAPI（后端）、SQLAlchemy ORM（数据库层）

## ADDED Requirements

### Requirement: 技术架构
系统 SHALL 采用前后端分离架构：
- 前端：React 18 + TypeScript + Ant Design 5 + React DnD（拖拽）
- 后端：Python FastAPI + SQLAlchemy 2.0 + Alembic（数据库迁移）
- 数据库：SQLite（默认）/ MySQL 双数据库支持，通过 SQLAlchemy ORM 统一抽象
- 部署：Docker + Docker Compose，包含前端容器、后端容器、可选 MySQL 容器
- 缓存：Redis（会话管理、排课锁）
- 文件存储：本地文件系统（Excel 导入导出、备份文件）

#### Scenario: Docker 部署
- **WHEN** 管理员执行 `docker-compose up -d`
- **THEN** 系统自动启动前端、后端、Redis 服务，默认使用 SQLite 数据库，系统可通过浏览器访问

#### Scenario: 数据库切换
- **WHEN** 超级管理员在系统设置中将数据库从 SQLite 切换为 MySQL
- **THEN** 系统提示需配置 MySQL 连接信息，确认后系统自动迁移数据至 MySQL，并重启后端服务

### Requirement: 多租户与权限体系
系统 SHALL 实现三级账户体系：

**超级管理员（Super Admin）**
- 管理所有学校账户的完整 CRUD
- 可修改学校账户的所有信息（含登录密码）
- 审核学校账户申请
- 审核学校账户的关键数据修改申请
- 系统设置（数据库切换、系统参数）
- 系统备份与还原
- 查看系统日志
- 仅超级管理员可见：数据库切换功能、日志系统、备份还原、全局管理

**学校账户（School Admin）**
- 管理本校基础数据（年级、班级、课程、教师、教室等）
- 管理本校教师账户
- 执行排课操作
- 修改部分自己的账户数据（名称、联系方式等）
- 关键数据修改（学校名称、地址等）需提交申请，经超级管理员批准后方可生效
- 可申请创建账户（需超级管理员审核）

**教师账户（Teacher）**
- 查看自己的课表
- 若被授予班主任权限，可查看并下载所管理班级的课表
- 修改自己的部分个人信息

#### Scenario: 学校账户申请
- **WHEN** 用户填写学校信息并提交账户申请
- **THEN** 系统创建待审核状态的学校账户，通知超级管理员，账户暂时无法登录

#### Scenario: 超级管理员审核通过
- **WHEN** 超级管理员审核通过学校账户申请
- **THEN** 学校账户激活，申请者收到通知并可登录系统

#### Scenario: 学校账户修改关键数据
- **WHEN** 学校账户修改学校名称等关键信息
- **THEN** 系统创建待审批记录，关键信息暂不更新，超级管理员审批后生效

#### Scenario: 班主任查看班级课表
- **WHEN** 教师账户被设置为某班班主任
- **THEN** 该教师登录后可查看并导出该班级的课表

### Requirement: 学期管理
系统 SHALL 支持自定义学期：

- 可创建、编辑、归档学期
- 每个学期包含：名称、起始日期、结束日期、总周数
- 每个学校可拥有多个学期，但仅一个"当前活跃学期"
- 学期课表独立保存，切换学期不影响其他学期数据
- 可复制历史学期的基础数据到新学期（教师、班级、课程等）

#### Scenario: 创建新学期
- **WHEN** 学校账户创建新学期并设为当前活跃学期
- **THEN** 系统切换到新学期上下文，所有排课操作基于新学期

#### Scenario: 保存学期课表
- **WHEN** 学校账户在排课界面点击保存
- **THEN** 当前课表数据保存至当前活跃学期，不影响其他学期数据

### Requirement: 时间段精细化控制
系统 SHALL 支持按年级精细化控制排课时间段：

**时间段类型**
- 早读：可设置节数（0-N节）、每节起止时间
- 正课：可设置节数（0-N节）、每节起止时间
- 晚自习：可设置节数（0-N节）、每节起止时间

**按年级控制**
- 每个年级可独立设置早读/正课/晚自习的节数和时间
- 每个年级可独立控制周一至周日每天的排课时段
- 可标记任意时段为"不排课"（如周三下午不排课、周六仅上半天等）

**时段模板**
- 提供常用时段模板（如"标准五天制"、"六天制"等）
- 可基于模板快速创建，再微调

#### Scenario: 设置年级时间段
- **WHEN** 学校账户为某年级设置周一至周五每天8节正课、1节早读、0节晚自习
- **THEN** 该年级的排课网格按此配置生成，排课操作受此时段约束

#### Scenario: 标记不排课时段
- **WHEN** 学校账户将某年级周三下午第7-8节标记为"不排课"
- **THEN** 一键排课和手动排课均不会在该时段安排课程，课表显示为空

### Requirement: 基础数据管理
系统 SHALL 支持以下基础数据的 CRUD 及 Excel 批量导入导出：

- **年级**：名称、排序
- **班级**：名称、所属年级、班主任（可选）
- **课程**：名称、代码、类型（必修/选修）、适用年级
- **教师**：姓名、工号、性别、联系电话、可授课程、所属教研组
- **教室**：名称、类型（普通教室/实验室/专用教室等）、容量、适用课程
- **教师-课程-班级关联**：哪位教师教哪门课对应哪个班

所有基础数据支持 Excel 模板下载、按模板填写后批量导入、批量导出为 Excel。

#### Scenario: Excel 批量导入教师
- **WHEN** 学校账户下载教师导入模板、填写数据后上传
- **THEN** 系统校验数据格式和完整性，校验通过后批量创建教师记录，校验失败则返回错误明细

#### Scenario: Excel 批量导出
- **WHEN** 学校账户点击导出按钮
- **THEN** 系统生成包含当前所有基础数据的 Excel 文件并下载

### Requirement: 流程式排课指引
系统 SHALL 提供流程式排课指引界面，步骤如下：

1. **学期设置**：创建/选择学期
2. **年级与时间段**：设置各年级的早读/正课/晚自习节数与时间
3. **基础数据**：录入/导入年级、班级、课程、教师、教室
4. **教学安排**：设置教师-课程-班级的授课关联
5. **排课规则**：设置排课约束（如某教师某时段不可排课、某课程不可排在某时段等）
6. **排课操作**：手动拖动排课 / 一键排课
7. **课表查看与导出**：查看、调整、导出课表

每一步在页面顶部以步骤条（Stepper）形式展示，用户可随时点击任意已完成步骤返回修改。修改基础数据后，系统提示是否需要重新排课。

#### Scenario: 流程导航
- **WHEN** 用户点击步骤条中的"基础数据"
- **THEN** 系统跳转到基础数据管理页面，用户可修改数据后返回当前步骤

#### Scenario: 修改已排课数据
- **WHEN** 用户修改了已排课的基础数据（如删除某教师）
- **THEN** 系统弹出提示"基础数据已变更，建议重新排课"，用户可选择立即重新排课或稍后处理

### Requirement: 排课操作
系统 SHALL 提供三种排课方式：

**1. 拖动排课**
- 课表以网格形式展示（行=时间段，列=周一至周日）
- 可将课程卡片拖动到目标单元格
- 拖动时实时检测冲突（教师冲突、教室冲突、班级冲突）
- 冲突时以红色标记并阻止放置，或以黄色警告提示（可配置）

**2. 一键排课**
- 基于排课规则和约束，使用约束满足算法自动生成课表
- 已手动固定的课程不受一键排课影响
- 一键排课完成后显示排课结果统计（成功率、冲突数等）

**3. 固定排课**
- 用户可手动将某节课固定到特定时段
- 固定的课程在课表上以锁定图标标识
- 一键排课不会调整已固定的课程
- 可随时取消固定

**冲突检测规则**
- 同一教师同一时段不可安排两门课
- 同一教室同一时段不可安排两门课
- 同一班级同一时段不可安排两门课
- 教师不可排课时段不可安排课程
- 不排课时段不可安排课程

#### Scenario: 拖动排课冲突
- **WHEN** 用户将课程A拖动到某时段，但该时段教师张老师已有课程B
- **THEN** 系统实时显示红色冲突提示"教师张老师在该时段已有课程B"，阻止放置

#### Scenario: 一键排课
- **WHEN** 用户点击一键排课
- **THEN** 系统基于约束算法自动排课，已固定课程保持不变，排课完成后显示统计信息

#### Scenario: 固定排课
- **WHEN** 用户将某课程固定到周一第1节
- **THEN** 该课程显示锁定图标，一键排课不会移动此课程

### Requirement: 课表查看与导出
系统 SHALL 支持多维度课表查看与导出：

**查看维度**
- 班级课表：按班级查看周课表
- 教师课表：按教师查看周课表
- 教室课表：按教室查看周课表
- 年级总课表：按年级查看所有班级课表汇总
- 全校总课表：查看全校课表汇总

**导出功能**
- 导出单个教师课表为 Excel/PDF
- 导出单个班级课表为 Excel/PDF
- 批量导出所有教师课表（每个教师一个 Sheet 或独立文件打包为 ZIP）
- 批量导出所有班级课表（每个班级一个 Sheet 或独立文件打包为 ZIP）
- 自定义导出：可选择导出范围（指定年级、指定教师、指定班级）、导出格式、导出内容（是否包含教室信息等）

#### Scenario: 批量导出教师课表
- **WHEN** 学校账户选择批量导出所有教师课表
- **THEN** 系统生成包含每位教师课表的 ZIP 文件下载

#### Scenario: 自定义导出
- **WHEN** 学校账户选择导出初一年级所有班级课表，格式为 Excel，包含教室信息
- **THEN** 系统按条件生成 Excel 文件，每个班级一个 Sheet，包含教室列

---

### Requirement: 排课规则与约束
系统 SHALL 支持以下排课约束设置：

- **教师不可排课时段**：指定教师在某天某时段不可排课（如教师张老师周三下午不排课）
- **课程时段限制**：指定课程不可排在某时段（如体育课不排在上午第1节）
- **课程连续性**：同一课程是否允许连排（如数学2节连排）
- **课程分散性**：同一课程是否要求分散在不同天（如语文不安排在同一天两节）
- **教室限制**：指定课程必须在特定类型教室上课（如化学课必须在实验室）
- **教师日课时上限**：每位教师每天最多排课节数
- **优先级**：约束可设置优先级，一键排课时按优先级处理冲突

#### Scenario: 设置教师不可排课时段
- **WHEN** 学校账户设置教师张老师周三下午不排课
- **THEN** 一键排课和手动排课均不会在周三下午为张老师安排课程

### Requirement: 日志系统
系统 SHALL 实现完善的日志记录：

- **操作日志**：记录所有用户的增删改操作（谁、何时、做了什么、影响范围）
- **登录日志**：记录所有登录尝试（成功/失败、IP、时间）
- **系统日志**：记录系统异常、错误、警告
- **排课日志**：记录排课操作的详细过程和结果

日志仅超级管理员可见，支持按时间、类型、用户、关键词筛选查询。

#### Scenario: 查看操作日志
- **WHEN** 超级管理员在日志页面选择时间范围和操作类型
- **THEN** 系统显示符合条件的日志列表，包含操作人、时间、操作内容、影响对象

#### Scenario: 定位系统问题
- **WHEN** 超级管理员搜索某错误关键词
- **THEN** 系统返回相关错误日志，包含堆栈信息，便于快速定位问题

### Requirement: 系统备份与还原
系统 SHALL 支持超级管理员进行系统级备份与还原：

- **备份**：一键备份当前数据库（含所有学校数据）、上传文件、配置信息
- **还原**：选择备份文件进行还原，还原前需确认操作
- **自动备份**：可配置定时自动备份（每日/每周）
- **备份管理**：查看备份列表、下载备份文件、删除旧备份

#### Scenario: 一键备份
- **WHEN** 超级管理员点击"立即备份"
- **THEN** 系统创建备份文件，包含数据库快照和配置，显示备份进度和结果

#### Scenario: 还原系统
- **WHEN** 超级管理员选择某备份文件点击"还原"
- **THEN** 系统弹出二次确认提示，确认后系统还原至备份时状态，所有当前连接断开并重新登录

### Requirement: 系统安全
系统 SHALL 实现以下安全措施：

- **认证**：JWT Token 认证，Token 有效期可配置
- **密码策略**：密码强度校验（最少8位，含大小写字母和数字）
- **会话管理**：支持单设备登录限制（可配置），超时自动登出
- **数据隔离**：不同学校数据严格隔离，学校账户仅可访问本校数据
- **SQL 注入防护**：使用 ORM 参数化查询
- **XSS 防护**：前端输入输出转义
- **CSRF 防护**：使用 CSRF Token
- **Rate Limiting**：API 请求频率限制
- **HTTPS**：生产环境强制 HTTPS

#### Scenario: 数据隔离
- **WHEN** 学校A账户请求获取教师列表
- **THEN** 系统仅返回学校A的教师数据，不包含其他学校的任何数据

#### Scenario: 并发访问
- **WHEN** 多个学校账户同时进行排课操作
- **THEN** 系统通过乐观锁和排课操作锁保证数据一致性，不出现数据错乱

---

### Requirement: 账户审批流程
系统 SHALL 实现以下审批流程：

**学校账户申请审批**
- 用户可在线填写学校信息申请账户
- 超级管理员在审批页面查看待审批列表
- 可批准或拒绝，拒绝时需填写原因
- 审批结果通知申请人（站内通知/邮件）

**学校关键数据修改审批**
- 学校账户修改关键信息（学校名称、地址等）时，创建审批单
- 超级管理员审批通过后，修改生效
- 审批期间，原数据正常使用

#### Scenario: 拒绝学校申请
- **WHEN** 超级管理员拒绝某学校账户申请并填写拒绝原因
- **THEN** 申请人收到拒绝通知及原因，账户保持不可用状态

---

### Requirement: 通知系统
系统 SHALL 实现站内通知功能：

- 审批结果通知
- 排课完成通知
- 系统维护通知（超级管理员发布）
- 通知支持已读/未读标记
- 通知列表分页展示

#### Scenario: 排课完成通知
- **WHEN** 一键排课完成
- **THEN** 系统向操作者发送站内通知"排课已完成，成功安排XX节，冲突XX节"

---

### Requirement: 系统设置
系统 SHALL 提供系统设置功能：

**超级管理员专属设置**
- 数据库类型切换（SQLite/MySQL）
- MySQL 连接配置
- 系统备份与还原
- 自动备份配置
- 系统维护模式开关
- 密码策略配置
- Token 有效期配置

**学校账户设置**
- 学校基本信息（部分需审批）
- 排课默认参数
- 课表显示偏好

#### Scenario: 切换数据库
- **WHEN** 超级管理员将数据库从 SQLite 切换为 MySQL 并填写连接信息
- **THEN** 系统测试连接，连接成功后执行数据迁移，迁移完成后切换到 MySQL

---

### Requirement: 数据库双引擎设计
系统 SHALL 兼容 SQLite 和 MySQL：

- 默认使用 SQLite，零配置启动
- 超级管理员可在设置中切换为 MySQL
- 使用 SQLAlchemy ORM 统一抽象，业务代码不直接编写 SQL
- 数据库切换时自动执行数据迁移
- Alembic 管理数据库 Schema 版本
- 仅超级管理员可见和操作此功能

#### Scenario: SQLite 默认启动
- **WHEN** 系统首次启动
- **THEN** 自动创建 SQLite 数据库文件，执行 Schema 迁移，系统可正常使用

### Requirement: 并发控制
系统 SHALL 支持多人同时访问和操作：

- 排课操作使用乐观锁，防止覆盖冲突
- 同一课表同时仅允许一人编辑（排课锁机制）
- 其他用户查看课表不受影响
- 使用 Redis 管理排课锁和会话

#### Scenario: 排课锁
- **WHEN** 用户A正在编辑某学期的课表
- **THEN** 用户B打开同一学期排课页面时，提示"当前课表正在被XX编辑，您仅可查看"

### Requirement: 界面设计
系统 SHALL 实现现代化美观界面：

- 使用 Ant Design 5 组件库，保持视觉一致性
- 响应式布局，适配桌面和平板
- 课表网格支持缩放和拖动
- 冲突单元格以颜色标识（红色=硬冲突、黄色=软冲突/警告）
- 固定课程以锁定图标标识
- 步骤条清晰指引排课流程
- 操作反馈及时（Toast 提示、加载动画）
- 深色/浅色主题切换

#### Scenario: 课表冲突可视化
- **WHEN** 课表中存在冲突
- **THEN** 冲突单元格以红色背景标识，鼠标悬停显示冲突详情

### Requirement: 可扩展性与易维护性
系统 SHALL 具备以下可扩展性和易维护性特性：

- 后端采用分层架构（Router → Service → Repository → Model）
- 排课算法与业务逻辑解耦，可替换算法实现
- 前端组件化开发，可复用
- API 版本化（/api/v1/）
- 配置外部化（环境变量 + 配置文件）
- Docker 镜像版本化管理
- 数据库迁移脚本版本化（Alembic）

### Requirement: 走班制支持
系统 SHALL 特别支持走班制排课：

- 教师固定在某教室上课，学生走班
- 支持设置"教学班"（走班组合），一个教学班可包含来自多个行政班的学生
- 教学班与行政班可并存
- 排课时以教学班为单位安排课程
- 行政班课表自动汇总其所属教学班的课程

#### Scenario: 创建走班教学班
- **WHEN** 学校账户创建一个"高一物理选考班"教学班，包含来自3个行政班的学生
- **THEN** 系统在排课时将此教学班作为一个排课单元，排课结果自动反映到各行政班课表

### Requirement: 课表调课
系统 SHALL 支持课表调整（调课）：

- 可对已排课表进行单节调课（换时段）
- 可进行两节课互换
- 调课时实时检测冲突
- 调课记录可追溯
- 支持批量调课（如某教师全天课程顺延一节）

#### Scenario: 单节调课
- **WHEN** 用户将周一第2节的数学课拖动到周一第3节
- **THEN** 系统检测冲突，无冲突则完成调课，有冲突则提示并阻止

#### Scenario: 两节课互换
- **WHEN** 用户选择周一第2节数学和周二第5节语文进行互换
- **THEN** 系统检测互换后是否冲突，无冲突则完成互换
